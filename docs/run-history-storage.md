# Run History Storage

The service can persist saved briefing runs to either local JSON files or Postgres.

## Storage Modes

Filesystem mode is the default:

```env
BRIEFING_RUN_STORE=filesystem
BRIEFING_RUN_DIR=runs
```

Postgres mode is opt-in:

```env
BRIEFING_RUN_STORE=postgres
DATABASE_URL=postgres://briefing_agent:briefing_agent@127.0.0.1:5432/briefing_agent
```

`POSTGRES_URL` is also accepted for environments that reserve `DATABASE_URL` for another service.

## Data Model

Saved briefings are written to `briefing_runs`:

- `id`: stable run id generated by the application
- `created_at`: application-side run timestamp
- `mode`: `mock` or `live`
- `topic` and `audience`: queryable request fields
- `brief_ids`: source briefing ids used by the agent
- `duration_ms`: end-to-end generation duration
- `request`, `result`, and `trace`: full JSONB payloads for audit and regression review
- `inserted_at`: database-side insert timestamp

The schema is documented in [`db/migrations/001_create_briefing_runs.sql`](../db/migrations/001_create_briefing_runs.sql). The application also runs `create table if not exists` before inserting so local demos do not require a separate migration step.

## Local Postgres Demo

Docker Compose starts a local Postgres database and configures the agent to save runs there:

```bash
docker compose up --build
```

Create a saved run:

```bash
curl --silent http://127.0.0.1:8787/brief \
  --header "content-type: application/json" \
  --data '{"topic":"Why evals matter for agent apps","audience":"engineering manager","saveRun":true}'
```

Inspect stored runs:

```bash
make postgres-runs
```

## Production Mapping

In a hosted deployment, `DATABASE_URL` would come from the platform secret manager and point to a managed Postgres instance. The local Kubernetes overlay intentionally remains filesystem-backed so it can run without adding a database dependency to local clusters.
